# 基于规则的推荐策略设计文档

> **项目背景**: H&M 个性化时尚推荐，137万用户 × 10.5万商品 × 3178万交易记录
> **设计原则**: 先跑通规则推荐基线，再谈模型优化。规则推荐是复杂模型的"兜底方案"和"评估基准"。
> **数据工具**: Python (pandas) 实现全部计算逻辑

---

## 一、推荐策略总览

```
用户打开APP
    │
    ├── 新用户 (无历史购买) ──→ 热门兜底策略 (全局热销TopN)
    │
    └── 老用户 (有历史购买) ──→ 用户偏好画像
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               品类偏好推荐    价格带偏好推荐    协同过滤推荐
                    │               │               │
                    └───────────────┴───────────────┘
                                    │
                           加权混合 (Ensemble)
                                    │
                           重排序 (业务规则)
                                    │
                           最终推荐 Top 20
```

**四种策略的关系**:

| 策略 | 解决什么问题 | 什么时候用 | 覆盖的用户 |
|------|-------------|-----------|-----------|
| 热门兜底 | 没有用户画像时的冷启动 | 新用户 / 数据不足 | 100% 用户兜底 |
| 品类偏好 | 知道你喜欢什么品类 → 推该品类热销商品 | 老用户，购买≥3次 | ~80% 用户 |
| 价格带偏好 | 知道你买什么价位 → 推同价位商品 | 价格偏好明显的用户 | ~90% 用户 |
| 协同过滤 | "买了A的人也买了B" | 有购买记录的所有用户 | ~99% 用户 |

## 二、用户偏好计算逻辑

### 2.1 品类偏好度

**业务含义**: 用户在某个品类的购买占比越高，说明越偏好该品类。

**SQL 逻辑**:

```sql
-- 计算每个用户在每个品类的购买占比
-- 目的: 知道用户的品类偏好分布，不是简单看"买最多的品类"
--        而是看"某个品类在用户购买中的占比是否显著高于人群均值"

WITH user_category_stats AS (
    -- Step 1: 用户在每个品类的购买次数
    SELECT
        t.customer_id,
        a.index_group_name,    -- 一级品类: Ladieswear / Menswear / Childrenswear
        a.index_name,          -- 二级品类
        a.section_name,        -- 三级品类
        COUNT(*) AS purchase_count,
        COUNT(DISTINCT t.article_id) AS unique_items
    FROM fact_transaction t
    JOIN dim_article a ON t.article_id = a.article_id
    WHERE t.data_split = 'train'
    GROUP BY t.customer_id, a.index_group_name, a.index_name, a.section_name
),
user_total AS (
    -- Step 2: 用户总购买次数
    SELECT customer_id, COUNT(*) AS total_purchases
    FROM fact_transaction WHERE data_split = 'train'
    GROUP BY customer_id
),
category_global AS (
    -- Step 3: 全局品类占比 (用于对比)
    SELECT
        index_group_name,
        index_name,
        section_name,
        COUNT(*) * 1.0 / SUM(COUNT(*)) OVER() AS global_ratio
    FROM fact_transaction t
    JOIN dim_article a ON t.article_id = a.article_id
    WHERE data_split = 'train'
    GROUP BY index_group_name, index_name, section_name
)

-- Step 4: 用户品类偏好 = 用户在该品类的占比 / 全局占比 > 1.0 说明偏好高于平均
SELECT
    u.customer_id,
    u.index_group_name,
    u.index_name,
    u.section_name,
    u.purchase_count,
    u.purchase_count * 1.0 / ut.total_purchases AS user_ratio,     -- 用户在该品类占比
    cg.global_ratio,                                                 -- 全局品类占比
    (u.purchase_count * 1.0 / ut.total_purchases) / cg.global_ratio AS preference_score, -- ★ 偏好得分
    -- preference_score > 1.0 = 比平均人群更偏好该品类
    -- preference_score > 2.0 = 显著偏好，可作为强信号
    u.unique_items
FROM user_category_stats u
JOIN user_total ut ON u.customer_id = ut.customer_id
JOIN category_global cg
    ON u.index_group_name = cg.index_group_name
   AND u.index_name = cg.index_name
   AND u.section_name = cg.section_name
WHERE ut.total_purchases >= 3  -- 至少买过3次，品类偏好才有统计意义
ORDER BY u.customer_id, preference_score DESC;
```

**为什么用 preference_score (用户占比/全局占比) 而不是直接用 purchase_count?**
- 举例：用户A买了100件东西，其中10件是Ladieswear → 占比10%
- 全局Ladieswear占比是50%
- 用户A的preference_score = 10%/50% = 0.2 → 实际上**不偏好**Ladieswear
- 如果只看purchase_count=10，你会误以为"买了10件，应该喜欢吧"
- 这种"相对于全局的偏好"才是做推荐时该用的信号

### 2.2 价格带偏好

**业务含义**: 用户习惯买什么价位，就推什么价位。

```sql
-- 价格带偏好计算
-- 思路: 
--   1. 先对全局价格分箱 (根据price分布P25/P50/P75)
--   2. 计算用户在各价格箱的购买占比
--   3. 最大占比的箱 = 用户的价格带偏好

WITH price_bounds AS (
    -- Step 1: 计算价格分布 (用训练集)
    SELECT
        -- 用PERCENTILE近似分箱边界
        -- 实际价格在0~0.59之间, 已做归一化
        0.015 AS p25_price,   -- 低端
        0.025 AS p50_price,   -- 中端
        0.040 AS p75_price    -- 高端
    -- 注: 实际执行时用PERCENTILE_CONT计算
),
user_price_band AS (
    -- Step 2: 用户每次购买落在哪个价格带
    SELECT
        t.customer_id,
        t.article_id,
        t.price,
        CASE
            WHEN t.price <= pb.p25_price THEN '低端'
            WHEN t.price <= pb.p50_price THEN '中端'
            WHEN t.price <= pb.p75_price THEN '中高端'
            ELSE '高端'
        END AS price_band
    FROM fact_transaction t
    CROSS JOIN price_bounds pb
    WHERE t.data_split = 'train'
)

-- Step 3: 每个用户最偏好的价格带
SELECT
    customer_id,
    price_band,
    COUNT(*) AS band_purchases,
    COUNT(*) * 1.0 / SUM(COUNT(*)) OVER(PARTITION BY customer_id) AS band_ratio
FROM user_price_band
GROUP BY customer_id, price_band;
```

### 2.3 颜色偏好与渠道偏好

```sql
-- 颜色偏好
SELECT
    t.customer_id,
    a.colour_group_name,
    COUNT(*) AS purchase_count,
    COUNT(*) * 1.0 / SUM(COUNT(*)) OVER(PARTITION BY t.customer_id) AS colour_ratio
FROM fact_transaction t
JOIN dim_article a ON t.article_id = a.article_id
WHERE t.data_split = 'train'
GROUP BY t.customer_id, a.colour_group_name;

-- 渠道偏好
SELECT
    customer_id,
    SUM(CASE WHEN sales_channel_id = 1 THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS online_ratio,
    CASE
        WHEN SUM(CASE WHEN sales_channel_id = 1 THEN 1 ELSE 0 END) * 1.0 / COUNT(*) >= 0.7 THEN '线上为主'
        WHEN SUM(CASE WHEN sales_channel_id = 1 THEN 1 ELSE 0 END) * 1.0 / COUNT(*) <= 0.3 THEN '线下为主'
        ELSE '混合渠道'
    END AS channel_preference
FROM fact_transaction
WHERE data_split = 'train'
GROUP BY customer_id;
```

## 三、商品相似度计算逻辑

### 3.1 基于品类层级的相似度

**思想**: 两个商品在品类树上的距离越近，它们越相似。

```
品类层级结构:
index_group > index > section > department > product_type > article

相似度规则:
  同一 product_type   → 相似度 = 1.0
  同一 department     → 相似度 = 0.8
  同一 section        → 相似度 = 0.6
  同一 index          → 相似度 = 0.4
  同一 index_group    → 相似度 = 0.2
  不同 index_group    → 相似度 = 0.0
```

```sql
-- 商品品类相似度矩阵 (两两商品之间)
-- 注意: 这是105,542 × 105,542的矩阵, 太大了, 不能全部计算
-- 策略: 不存全量矩阵, 只在推荐时"对用户买过的商品"找同品类商品

-- 推荐时实时计算: 找到和用户历史购买商品同品类的最热销商品
SELECT DISTINCT
    a2.article_id AS candidate_article_id,
    a2.prod_name,
    a2.department_name,
    -- 品类相似度得分
    CASE
        WHEN a1.product_type_no = a2.product_type_no   THEN 1.0
        WHEN a1.department_no  = a2.department_no      THEN 0.8
        WHEN a1.section_no     = a2.section_no         THEN 0.6
        WHEN a1.index_code     = a2.index_code         THEN 0.4
        WHEN a1.index_group_no = a2.index_group_no     THEN 0.2
        ELSE 0.0
    END AS category_similarity,
    fas.total_purchases AS popularity  -- 同品类内按热度排序
FROM dim_article a1
JOIN dim_article a2
    ON a1.index_group_no = a2.index_group_no  -- 至少同一大类
   AND a1.article_id <> a2.article_id
JOIN feature_article_stats fas ON a2.article_id = fas.article_id
WHERE a1.article_id = '目标商品'  -- 用户历史购买的商品
ORDER BY category_similarity DESC, popularity DESC
LIMIT 100;
```

### 3.2 基于共购的相似度 (协同过滤)

**思想**: 如果很多用户同时买了商品A和商品B，说明A和B之间存在关联。

```sql
-- 共购相似度 = Jaccard系数
-- sim(A,B) = |买了A的用户 ∩ 买了B的用户| / |买了A的用户 ∪ 买了B的用户|
-- 取值范围: 0 ~ 1, 越大越相似

WITH article_users AS (
    -- Step 1: 每个商品被哪些用户购买过
    SELECT article_id, customer_id
    FROM fact_transaction
    WHERE data_split = 'train'
    GROUP BY article_id, customer_id
),
co_purchase AS (
    -- Step 2: 计算任意两个商品的共购用户数
    SELECT
        a1.article_id AS article_a,
        a2.article_id AS article_b,
        COUNT(DISTINCT a1.customer_id) AS co_buyers,  -- 交集: 同时买了A和B的人数
        -- 注意: 效率很低, 实际只计算"热门商品之间"或"用户最近购买的商品"
        (SELECT COUNT(DISTINCT customer_id) FROM article_users WHERE article_id = a1.article_id) AS buyers_a,  -- 买了A的人数
        (SELECT COUNT(DISTINCT customer_id) FROM article_users WHERE article_id = a2.article_id) AS buyers_b   -- 买了B的人数
    FROM article_users a1
    JOIN article_users a2
        ON a1.customer_id = a2.customer_id
       AND a1.article_id < a2.article_id   -- 去重, A<B避免重复计算
    GROUP BY a1.article_id, a2.article_id
    HAVING COUNT(DISTINCT a1.customer_id) >= 5  -- 至少有5个共同购买者才考虑
)

-- Step 3: 计算Jaccard相似度
SELECT
    article_a,
    article_b,
    co_buyers,
    buyers_a,
    buyers_b,
    -- ★ Jaccard = 交集 / 并集 = co_buyers / (buyers_a + buyers_b - co_buyers)
    co_buyers * 1.0 / (buyers_a + buyers_b - co_buyers) AS jaccard_similarity,
    -- ★ Cosine = co_buyers / sqrt(buyers_a * buyers_b)  (另一种算法)
    co_buyers * 1.0 / SQRT(buyers_a * buyers_b) AS cosine_similarity
FROM co_purchase
WHERE buyers_a >= 20 AND buyers_b >= 20  -- 过滤冷启动噪音
ORDER BY jaccard_similarity DESC;
```

### 3.3 颜色相似度

```sql
-- 颜色相似度: 同色系=1.0, 互补色=0.5, 其他=0.0
-- 简化版: 直接用 perceived_colour_master_name 匹配

SELECT
    a1.article_id AS source_article,
    a2.article_id AS similar_article,
    a2.prod_name,
    -- 颜色相似度
    CASE
        WHEN a1.colour_group_code = a2.colour_group_code             THEN 1.0  -- 同颜色组
        WHEN a1.perceived_colour_master_id = a2.perceived_colour_master_id THEN 0.8  -- 同主色
        WHEN a1.perceived_colour_value_id = a2.perceived_colour_value_id THEN 0.3  -- 同色值(如都深色)
        ELSE 0.0
    END AS colour_similarity
FROM dim_article a1
CROSS JOIN dim_article a2
WHERE a1.article_id = '目标商品'
  AND a1.article_id <> a2.article_id
ORDER BY colour_similarity DESC
LIMIT 50;
```

## 四、推荐规则

### 4.1 推荐规则决策树 (业务逻辑)

```
输入: customer_id
输出: 推荐商品列表 Top 20

┌─ 该用户是否有购买记录?
│
├─ 否 (新用户 / 冷启动) ──→ 策略0: 全局热销 Top 20
│                           规则: 按全局购买次数降序
│                           过滤: 过去30天有库存的商品
│                           兜底: 如果热门不够20个,随机补全
│
└─ 是 (有购买记录)
    │
    ├─ 购买次数 ≤ 2 (行为太少,画像不可靠)
    │    ──→ 策略1: 热门推荐 60% + 随机探索 40%
    │
    └─ 购买次数 ≥ 3 (画像可信)
         │
         ├─ 策略2: 品类偏好推荐 (占40%推荐位 = 8个)
         │    规则: 取用户Top3偏好品类, 每个品类推该品类下热销商品
         │    偏好品类 = preference_score > 1.0 且 取前3
         │    排序: 品类相似度 DESC, 然后销量 DESC
         │
         ├─ 策略3: 价格带推荐 (占25%推荐位 = 5个)
         │    规则: 匹配用户偏好价格带 ± 1档
         │    排序: 该价格带内销量 DESC
         │
         ├─ 策略4: 协同过滤推荐 (占25%推荐位 = 5个)
         │    规则: 取用户最近购买的5个商品
         │          对每个商品找Top5共购相似商品
         │          合并去重, 按jaccard DESC排序
         │
         └─ 策略5: 颜色偏好推荐 (占10%推荐位 = 2个)
              规则: 用户最偏好的颜色组 + 同品类热销
              作用: 增加多样性,避免全是"猜你喜欢"

最终: 合并4个策略的结果 → 去重 → 打散(同department不超过3个) → 截断Top 20
```

### 4.2 Excel 伪代码实现

以下伪代码可以直接在 Excel 或 Python 中实现:

```
============================================================
策略0: 热门兜底推荐
============================================================
INPUT: 无 (不需要用户画像)
PROCESS:
    1. 统计每个 article_id 的 purchase_count (全局)
    2. 过滤: popularity_tier != '冷启动物品'  (排除只被买1-3次的)
    3. 降序排列
    4. 取 Top 20
OUTPUT: [article_1, article_2, ..., article_20]

============================================================
策略2: 品类偏好推荐 (核心策略,权重最高)
============================================================
INPUT: customer_id
PROCESS:
    1. 查该用户在各 index_name 的 purchase_count
    2. 计算 user_ratio = purchase_count / 用户总购买次数
    3. 计算 preference_score = user_ratio / 该品类的 global_ratio
    4. 筛选 preference_score > 1.0 的品类, 按 preference_score 降序取前3
    5. 对每个偏好品类:
       a. 找 dim_article 中同品类的所有商品
       b. 排除该用户已经购买过的商品
       c. 按 global_purchase_count 降序
       d. 取 Top 5
    6. 合并3个品类的结果(最多15个)
    7. 如果不够, 用热门商品补全
OUTPUT: [article_1, ..., article_8]  (8个推荐位)

============================================================
策略4: 协同过滤推荐
============================================================
INPUT: customer_id
PROCESS:
    1. 查该用户最近购买的5个商品 (按 t_dat 降序)
    2. 对每个商品:
       a. 在共购相似度表中查 Top 5 相似商品 (jaccard > 0.05)
       b. 排除用户已购买的商品
    3. 合并所有相似商品, 去重
    4. 按 jaccard_similarity 加权求和 (同一个商品被多个source推荐则累加得分)
    5. 降序排列, 取 Top 5
OUTPUT: [article_1, ..., article_5]  (5个推荐位)

============================================================
最终重排规则 (对所有策略结果统一处理)
============================================================
INPUT: 策略0~5的合并推荐列表 (去重后可能有40-60个候选)
PROCESS:
    规则1 (品类打散): 同一 department_name 最多出现3个, 第4个起降到最后
    规则2 (冷启动加权): 如果 popularity_tier='冷启动物品', 排序位置+2 (向上提2位)
    规则3 (去已购): 移除用户过去30天内购买过的商品
    规则4 (价格多样性): 确保至少有2个价格带的商品 (高/中/低各至少1个)
    规则5 (截断): 取 Top 20
OUTPUT: 最终推荐列表 [article_1, ..., article_20]
```

### 4.3 推荐规则汇总表

| 策略编号 | 策略名称 | 适用条件 | 推荐位占比 | 核心逻辑 | 可调参数 |
|----------|----------|----------|-----------|---------|---------|
| 0 | 热门兜底 | 新用户/无购买 | 100% (兜底) | 全局销量降序 | 时间窗口 |
| 1 | 混合探索 | 购买≤2次 | 60%热门+40%随机 | 热门为主+探索 | 热门/随机比例 |

| 5 | 颜色偏好 | 购买≥3次 | 10% (2位) | 颜色组匹配 | 取前几个颜色 |

## 五、A/B 测试评估方案

### 5.1 实验设计

```
┌─────────────────────────────────────────────────────┐
│                    实验框架                           │
│                                                      │
│  时间: 14天 (覆盖两个完整周,消除周末效应)               │
│  流量: 100%                                          │
│  ┌──────────────┐  ┌──────────────┐                  │
│  │ 对照组 A (50%)│  │ 实验组 B (50%)│                 │
│  │              │  │              │                  │
│  │ 热门推荐兜底  │  │ 基于规则推荐  │                  │
│  │ (策略0)      │  │ (策略2+3+4+5) │                 │
│  └──────────────┘  └──────────────┘                  │
│         │                  │                          │
│         └────────┬─────────┘                          │
│                  ▼                                    │
│          对比核心指标                                  │
│          显著性检验 (t检验)                            │
│          p < 0.05 → 显著有效                          │
└─────────────────────────────────────────────────────┘

分流方式: customer_id 哈希值取模
    hash(customer_id) % 2 == 0 → 对照组A
    hash(customer_id) % 2 == 1 → 实验组B
```

### 5.2 评估指标体系

| 层级 | 指标 | 计算公式 | 含义 | 优先级 |
|------|------|----------|------|--------|
| **北极星** | 人均GMV | SUM(price) / COUNT(DISTINCT user) | 每个用户贡献的GMV | ⭐⭐⭐ |
| **核心** | 点击率(CTR) | 点击次数 / 曝光次数 | 推荐有没有吸引注意 | ⭐⭐⭐ |
| **核心** | 购买转化率(CVR) | 购买次数 / 曝光次数 | 推荐有没有促成购买 | ⭐⭐⭐ |
| **观察** | 推荐覆盖率 | 被推荐过的商品数 / 总商品数 | 多少商品有曝光机会 | ⭐⭐ |
| **观察** | 品类多样性 | 推荐列表中不同department的数量 | 推荐是否多样化 | ⭐⭐ |
| **观察** | 冷启动物品曝光占比 | 冷启动商品曝光 / 总曝光 | 新品有没有被看到 | ⭐⭐ |
| **观察** | 日均活跃度 | 有交易行为的用户数 / 总用户数 | 用户是否更活跃了 | ⭐ |
| **诊断** | 客单价 | GMV / 购买订单数 | 用户是不是买了更贵的 | ⭐ |

### 5.4 判断标准

| 场景 | 判断 | 行动 |
|------|------|------|
| p < 0.05, 提升 > 5% | ✅ 策略有效 | 全量上线该策略 |
| p < 0.05, 提升 0~5% | ⚠️ 微弱有效 | 优化策略后再测 |
| p < 0.05, 提升 < 0 | ❌ 策略有害 | 立即下线,排查原因 |
| p >= 0.05 | ❓ 无法判断 | 延长实验时间或加大样本量 |

### 5.5 后续: 多臂老虎机 (MAB) 上线后持续优化

```
如果用 Thompson Sampling:
  每个策略 = 一个"臂"
  臂的表现 = Beta分布(α=购买次数+1, β=未购买次数+1)
  每次推荐时: 从每个臂的Beta分布采样 → 选采样值最大的策略
  
效果: 
  - 好策略自动获得更多流量
  - 差策略自动减少流量
  - 不需要人工干预
```
