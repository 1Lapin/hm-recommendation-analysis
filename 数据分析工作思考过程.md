# 数据分析工作思考过程 — 推荐系统项目中的数据分析角色

> **写给小组同学**：这份文档梳理了"作为数据分析师，我在推荐系统项目的每一层中能够贡献什么"，以及背后的思考逻辑。推荐系统不是算法工程师一个人的事，数据分析从头到尾都在发挥关键作用。

---

## 一、先想清楚：数据分析在这个项目里的定位

### 1.1 我的角色不是"跑数的"

很多同学觉得数据分析就是：
```
别人提需求 → 我写SQL → 出张表 → 交差
```

但在这个项目里，数据分析要做的是：
```
理解业务问题 → 探查数据能回答什么 → 设计分析框架
→ 用数据验证假设 → 把结论翻译成"下一步该做什么"
→ 推动决策落地
```

### 1.2 一个贯穿始终的问题

> **"我做这一步分析，最终影响了什么推荐决策？"**

如果回答不上来，那这一步分析可能不需要做。
如果回答得上来，那报告里第一句就应该写这个。

举个例子：

| 分析任务 | 影响了什么决策 |
|----------|---------------|
| 统计每个品类的购买分布 | 决定召回层要不要做品类过滤 |
| 计算用户平均购买间隔 | 决定训练集应该回溯多长时间 |
| 计算共购相似度阈值 | 决定协同过滤最少要有几个共同购买者才保留 |
| 统计冷启动商品数量 | 决定重排层新品加权比例 |

## 二、按推荐系统层级，逐层拆解数据分析的工作

### 2.1 数据层 — 这是数据分析的主战场

**我的思考起点**：拿到4个CSV文件，我先问自己三个问题：

> **Q1: 这个数据"能"支撑什么样的推荐系统？**
>
> 答：有交易记录 → 可以做行为推荐；有商品属性 → 可以做内容推荐；有时间戳 → 可以做序列推荐。但是——没有用户显式评分、没有商品图片（只有文本描述）、没有点击日志（只有购买）。所以这是一个"隐式反馈 + 内容属性"的场景，适合协同过滤 + 内容召回的组合。

> **Q2: 这个数据的"天花板"在哪里？**
>
> 答：价格是归一化值（不是真实价格），无法直接用于业务报表。客户ID和邮编是哈希的，无法关联外部数据。交易只有购买没有浏览，意味着我们只能优化"买不买"、"买什么"，无法优化"看不看"。这些是数据自身的限制，不是技术问题。

> **Q3: 哪些数据问题会直接影响推荐效果？**
>
> 答：
> - 14%的商品购买≤5次 → 冷启动问题，召回层和重排层都要特殊处理
> - 65%的用户Active为空 → 不能把Active当特征用，必须用交易行为反推
> - fashion_news_frequency 出现"None"和"NONE" → 特征编码会分裂，必须标准化
> - 9699个用户无任何交易 → 新用户冷启动，必须有热门兜底策略

**我给小组的输出**：
- 数据质量报告（缺失率、分布、异常值）
- 特征可行性评估（哪些字段能做特征、哪些不能、为什么）
- 数据清洗脚本 + 清洗前后的对比

**我的思考框架**：

```
原始数据
    │
    ├─ 完整性检查: 缺了多少? 缺失是随机的还是结构性的?
    │   → 结构性缺失(如Active跟会员类型相关) → 不能简单删除
    │   → 随机缺失 → 可删除或用均值填充
    │
    ├─ 一致性检查: 同一信息是否有多种表达?
    │   → "None" != "NONE" → 标准化
    │
    ├─ 合理性检查: 数据是否符合业务逻辑?
    │   → 99.3%用户有交易但66%标记为不活跃 → 数据之间的逻辑矛盾
    │   → 必须用交易行为修正Active字段
    │
    └─ 可用性判断: 哪些字段能真正用于推荐?
        → 哈希后的postal_code → 不能用于地理推荐
        → detail_desc虽然是文本但416行空值 → 需要填充后使用
```

### 2.2 特征工程层 — 数据分析定义"什么是好的特征"

**我的思考起点**：给模型喂什么特征，不是拍脑袋决定的。

> **核心方法论: 特征 = 业务理解 + 数据验证**

**品类偏好特征的设计过程**：

```
Step 1: 业务直觉
  "用户应该有品类偏好，比如常买女装的人不太可能突然买男装"
  
Step 2: 数据验证
  → 跑SQL: 每个用户买了多少个不同品类?
  → 结果: P50用户只买过5个不同department, P90买过12个
  → 结论: 品类偏好确实存在, 且差异很大

Step 3: 特征设计
  → 方案A: 直接用 purchase_count → ❌ 买最多的品类不等于偏好
  → 方案B: 用 user_ratio (用户占比) → ❌ 没有参照物
  → 方案C: 用 user_ratio / global_ratio → ✅ 相对偏好得分
          ↑ 这个设计来自一个洞察: 
            "如果全球50%的人买Ladieswear, 而用户只买了30%,
             那其实说明这个用户不那么偏好女装"
  
Step 4: 验证特征有效性
  → 计算所有用户的Top1偏好品类
  → 看该品类是否确实是用户最近购买的品类
  → 准确率 = ? 如果太低, 说明特征设计有问题
```

**年龄特征的处理思考**：

```
原始数据: age字段, 16~99岁, 1.2%缺失

问题: 年龄应该作为数值(16岁)还是分类(16-24)?
思考:
  - 数值: 模型可以学到"年龄越小越偏好某些品类", 但假设是线性关系
  - 分类: 允许非线性的偏好差异(18岁和22岁偏好可能完全不同)
  - 实际上服装偏好和年龄的关系肯定是非线性的
  - 所以做年龄段分桶更合理

怎么分桶?
  → 先看数据: min=16, max=99, avg=36.4
  → 再看看年龄和品类的交叉表:
    18-24岁买什么最多? → Basic系列
    45-54岁买什么最多? → ?
  → 基于交叉表的结果来定分桶边界, 而不是机械地等距切分
  → 这就是"用数据驱动特征设计, 而不是凭直觉"
```

**我给小组的输出**：
- 每个特征的"诞生过程"（业务直觉 → 数据验证 → 设计方案 → 效果评估）
- 特征重要性排序（训练模型后用SHAP或feature_importance验证）
- 特征字典文档（字段名、含义、来源、取值范围、缺失处理方式）

### 2.3 召回层 — 数据分析评估"召回得够不够"

**我的思考起点**：算法同学做了双塔模型 + FAISS 检索。数据分析要回答的问题不一样——

> **不是"模型AUC多少"，而是"该召回的商品都召回来了吗？"**

**召回命中率分析**：

```
分析逻辑:
  1. 拿验证集用户的真实购买商品
  2. 看这些商品是否在召回层的200个候选里
  3. 如果不在，召回的200个里有什么?

关键结论:
  ✓ 热销商品: 命中率高 (因为双塔模型学了热度特征)
  ✗ 长尾商品: 命中率低 (用户只买过1-2次的品类, 模型没学到)
  
→ 数据分析的建议: 召回层需要补充"品类规则召回"通道
  → 如果用户过去买过某品类, 该品类下热销商品强制进入候选集
  → 这就是从"模型召回"到"模型+规则混合召回"的升级
```

**召回覆盖度分析**：

```
分析逻辑:
  统计每个用户的召回候选集里有多少个不同的department
  → P50 = 3个department? → 太集中了, 推荐列表会很单调
  → P50 = 10个department? → 多样性OK

→ 如果覆盖度太低, 建议在召回层加一个"多样性召回"通道
  (随机采样不同department的热门商品, 混入候选集)
```

**我给小组的输出**：
- 召回命中率报告（热门 vs 长尾的命中率差异）
- 召回覆盖度分析（用户维度、品类维度）
- 召回通道贡献分析（哪个召回通道贡献了多少最终购买）

### 2.4 排序层 — 数据分析定义"什么叫好"

**我的思考起点**：排序模型用了多目标（CTR + CVR + ATC）。数据分析负责定义"这些目标的权重怎么设"。

> **问题: CTR权重0.3, CVR权重0.5, ATC权重0.2 —— 这个数字从哪来？**

**权重设定的分析逻辑**：

```
方法: 模拟不同权重组合下的推荐列表, 看对最终GMV的影响

做一张表格 (在Excel里):
  权重组合          Top20推荐列表的预期GMV    覆盖率   冷启动占比
  CTR=0.5/CVR=0.3  → 高点击低转化 → GMV偏低   ↑         ↓
  CTR=0.3/CVR=0.5  → 中点击高转化 → GMV最高   →         →
  CTR=0.1/CVR=0.7  → 低点击高转化 → GMV偏低   ↓         ↓

→ 结论: CTR=0.3/CVR=0.5 的GMV最高
→ 但这是在历史数据上算的, 线上可能有差异
→ 所以A/B测试时应该同时对比2-3组权重
```

**特征贡献度分析**：

```
问题: 哪些特征对最终排序影响最大?
  
方法: 训练一个简单的LightGBM排序模型
  → 输出特征重要性
  → 通常发现: RFM特征 > 品类特征 > 价格特征 > 颜色特征
  
→ 给小组的结论: 
  "时间特征(最近购买)和频率特征(购买次数)是最重要的,
   所以后续特征工程应该重点挖掘时间序列特征,
   比如'最近3次购买的价格趋势'、'购买间隔在变长还是变短'"
```

**我给小组的输出**：
- 多目标权重敏感性分析
- 特征重要性报告 + 特征贡献度归因
- 排序模型离线评估 (NDCG@20 / MRR)

---

### 2.5 重排层 — 数据分析做"业务翻译官"

**我的思考起点**：重排层全是业务规则。数据分析的角色是把"业务想要什么"翻译成"可以量化的规则"。

> **"运营说要多推新品" → 数据分析要回答: 推多少? 加权多少? 会不会影响GMV?**

**新品加权比例的确定**：

```
问题: 冷启动物品加权多少合适?

分析逻辑:
  1. 对历史数据模拟: 给冷启动商品分别加权 0% / 5% / 10% / 15% / 20%
  2. 看对Top 20推荐列表的影响:
     加权0%:  冷启动商品 0.0个/20 → GMV = 基线
     加权5%:  冷启动商品 0.3个/20 → GMV = 基线×99.8%
     加权10%: 冷启动商品 0.8个/20 → GMV = 基线×99.2%
     加权15%: 冷启动商品 1.5个/20 → GMV = 基线×97.5%
     加权20%: 冷启动商品 3.2个/20 → GMV = 基线×94.0%
  
  3. 业务决策:
     "我们愿意牺牲多少GMV来换取新品曝光?"
     → 如果目标是"GMV下降不超过2%", 则加权≤15%
     → 推荐设置: 冷启动加权 = 15%
```

**品类打散力度的确定**：

```
问题: "同一品类最多出现3个" —— 为什么是3? 不是5? 不是2?

分析逻辑:
  1. 看用户实际购买行为的品类集中度:
     P50用户: 最近10次购买覆盖 4 个department
     P90用户: 最近10次购买覆盖 7 个department
     
  2. 结论: 用户自然行为就有品类多样性
     推荐列表不应该更集中, 至少保持同等水平的多样性
  
  3. 推20个商品, 如果用户平均会看4个department
     → 每个department平均5个, 但考虑头部偏好
     → 上限设3个合理: 允许偏好品类多推, 但超过3个则强制多样化

→ 这个"3"不是拍脑袋的, 是从用户实际行为的分布中算出来的
```

**我给小组的输出**：
- 重排规则参数敏感性分析（每个参数的改动对最终GMV的影响）
- 重排规则的"模拟沙盘"（改动参数 → 看推荐列表怎么变 → 看预期效果）

### 2.6 A/B测试层 — 数据分析的核心战场

**我的思考起点**：A/B测试不只是"跑个t检验"。整个实验的设计、执行、解读都是数据分析的工作。

> **最容易犯的错误: 实验还没跑完就下结论。**

**实验设计的思考链**：

```
问题1: 需要多少样本?
  → 先估计效果量(预期提升5% GMV)
  → 算Power Analysis: 每组需要多少用户才能以80%的统计功效检测到5%的提升?
  → 如果样本不够 → 延长实验时间

问题2: 怎么分流?
  → 随机分流 → 用customer_id哈希 → 确保用户粘在同一个组(不会同一个用户今天看A明天看B)
  → 如果分流不均 → 先做AA测试验证(两组都用同一个策略, 看指标有没有差异)

问题3: 跑多久?
  → 至少14天 → 覆盖两个完整周 → 消除周末效应
  → 还要考虑"新奇效应": 新推荐刚上的前几天, 用户因为新鲜感点击偏高
  → 所以分析时剔除前2天数据

问题4: 为什么p<0.05但我不一定信?
  → 如果同时看10个指标, 会有5%×10=50%的概率至少1个碰巧显著
  → 解决方案: Bonferroni校正 (0.05/10=0.005)
  → 或者只把CTR、CVR、GMV作为主指标, 其他作为观察指标
```

**实验结果的解读**：

```
场景1: CTR上升了, CVR没变, GMV没变
  → 解释: 推荐更吸引眼球了, 但推荐的跟用户想要的还是不一样
  → 行动: 重点优化CVR, 可能是价格带没匹配好

场景2: CTR没变, CVR上升了, GMV上升了
  → 解释: 推荐的商品"更准了", 点进来的人更容易下单
  → 行动: 这是最好的情况, 放大策略

场景3: CTR/CVR都上升, 但留存下降
  → 解释: 推荐"太好"了 → 用户很快买到想买的 → 走了
  → 或者: 推荐太单一 → 用户腻了
  → 行动: 增加多样性, 加入"浏览型推荐"和"发现型推荐"

→ 好的数据分析不只是报告数字, 而是解释"为什么"并建议"怎么做"
```

**我给小组的输出**：
- 实验设计文档（分流方案、样本量计算、实验时长）
- 每日实验监控看板（各指标趋势、显著性变化）
- 实验结论报告（含统计检验 + 业务解读 + 行动建议）

## 三、数据分析在项目各阶段的交付物地图

| 项目阶段 | 数据分析的交付物 | 解决了什么问题 | 给谁用 |

| 持续优化 | 月度推荐效果分析 | 追踪效果, 发现退化信号 | 全组 |

## 四、我作为数据分析师的核心竞争力提炼

### 4.1 三个核心能力

```
1. 把业务问题翻译成数据问题
   运营说"推荐不准" → 数据分析问: "不准指的是什么?"
   → 点击率低? 买了又退? 用户看了但没买?
   → 不同的"不准"对应不同的数据分析方案

2. 把数据结论翻译成业务行动
   数据显示"价格偏好特征重要性低" → 数据分析说:
   "不是因为价格不重要, 而是用户的价格偏好很稳定,
    模型早就'学会'了, 不需要反复强调。
    但冷启动阶段价格偏好很重要, 应该前3次购买就锁定。"

3. 设计"够用"的分析方案
   不做过度分析:
   - 10万商品的两两相似度矩阵是100亿个组合, 算不完也没必要
   - 方案: 只算"热门商品之间"和"用户最近购买的商品之间" → 缩小到百万级
   - 这是数据分析的"工程思维"
```

### 4.2 我和算法工程师的协作边界

| 任务 | 数据分析做 | 算法工程师做 | 我们一起做 |
|------|-----------|-------------|-----------|
| 数据清洗 | ✅ 主责 | — | — |
| 特征定义 | ✅ 主责 | 提需求 | 特征评审 |
| 模型训练 | 提供训练数据 | ✅ 主责 | — |
| 模型评估 | ✅ 离线评估 | ✅ 在线评估 | 综合判断 |
| 策略设计 | ✅ 规则策略 | ✅ 模型策略 | 混合方案 |
| A/B测试 | ✅ 实验设计+分析 | 工程实现 | 结果解读 |
| 业务沟通 | ✅ 主责 | — | — |

### 4.3 如果只能记住三句话

> **第一句**: 每一个分析都要能回答"这个结论影响了什么推荐决策"。
> 回答不了 = 这个分析可能不需要做。

> **第二句**: 数据会告诉你它自己的局限——在开始建模之前，先把数据的能力边界摸清楚。
> 能做什么、不能做什么，比做了多少更重要。

> **第三句**: 推荐系统的最终裁判是A/B测试，不是离线指标。
> 离线好的模型上线不一定好，但离线差的模型上线一定差。

## 五、项目的"数据分析思维"总结

```
这个项目教会我们的不是"怎么用pandas写循环"，
而是"拿到一个数据集到做出一个推荐系统，中间每个决策点是什么"：

数据层:    这个数据能支撑什么? 天花板在哪? 坑在哪?
特征层:    每个特征的设计有数据支撑吗? 不是"我觉得应该有用"而是"数据说有用"
召回层:    召回的不只要"准", 还要"全"——该召回的都没漏, 才是好召回
排序层:    权重不是拍脑袋, 是做敏感性分析得出来的最优组合
重排层:    每个规则参数都对应一个"GMV vs 业务目标"的权衡
A/B层:    统计显著 ≠ 业务显著, p值是判断工具不是最终结论

最终目标: 用数据驱动推荐策略优化, 最终提升GMV和用户留存
```
