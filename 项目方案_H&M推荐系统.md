# H&M 个性化时尚推荐系统 — 项目整体方案

> **项目定位**: 数据驱动的推荐策略优化，最终提升 GMV（商品交易总额）和用户留存
> **数据规模**: 137万用户 × 10.5万商品 × 3178万条交易记录
> **时间跨度**: 2018-09-20 ~ 2020-09-22（2年）

---

## 一、项目要解决什么问题

### 1.1 业务问题

H&M 作为全球快时尚品牌，面临三个核心挑战：

| 问题 | 现状 | 对业务的影响 |
|------|------|-------------|
| **商品长尾严重** | 14%的商品仅被购买≤5次，反而占库存大头 | 库存积压 → 打折清仓 → 利润被吃掉 |
| **用户留存不稳定** | 平均购买间隔大，9千+用户完全无交易 | 拉新成本是留存的5倍，流失就是亏钱 |
| **推荐缺乏个性** | 如果没有推荐系统，所有人看到一样的首页 | 点击率低 → 转化低 → GMV增长乏力 |

### 1.2 一个具体的场景

> 用户 A（25岁女性，过去常买黑色基础款，线上渠道为主）打开 H&M APP。
>
> **没有推荐系统**: 首页推了当季爆款——男士夹克。用户A划走，关闭APP。
> **有推荐系统**: 召回层从10万商品里筛出200个候选 → 排序层根据她的历史偏好（女装、黑色、基础款、线上）打分排序 → 重排层加入新品推广、保证多样性 → 最终展示20个商品。用户A点进了"黑色罗纹吊带背心"，购买了。
>
> **差距**: 0 购买 vs 1 购买。放大到137万用户，这就是 GMV 的来源。

### 1.3 核心思想

```
数据 → 清洗 → 特征 → 召回(粗筛) → 排序(精排) → 重排(业务规则) → 推荐结果
                                                    ↑
                                              A/B测试验证效果
```

三个关键词:
- **召回层**: 从10万商品里快速筛出200个候选（管"有没有"）
- **排序层**: 对200个候选按用户偏好打分排序（管"准不准"）
- **重排层**: 加入库存、新品、多样性等业务规则（管"赚不赚钱"）

## 三、各层技术方案详解

### 3.1 数据层 — 已完成 ✅

| 产出 | 内容 | 工具 |
|------|------|------|
| 7个 SQL 脚本 | 建表→清洗→特征→验证→Tableau视图 | Navicat / MySQL |
| dim_customer | 137万用户清洗后的维度表 | — |
| dim_article | 10.5万商品维度表（含品类路径） | — |
| fact_transaction | 3178万交易事实表（含训练/验证划分） | — |
| feature_user_rfm | 用户 RFM 行为特征宽表 | — |
| feature_article_stats | 商品协同特征宽表 | — |

### 3.2 特征工程层 — 待实现

| 步骤 | 技术 | 输出 |
|------|------|------|
| 文本特征 | jieba分词 + TF-IDF / BERT | 商品描述 → 文本向量 |
| 类别特征 | Label Encoding / Embedding | 品类层级 → 低维稠密向量 |
| 数值特征 | Min-Max归一化 / 分桶 | age → 年龄段, price → 价格段 |
| 序列特征 | 按时间排序用户购买序列 | 用户兴趣演化序列 |

### 3.3 召回层 — 待实现

**为什么选双塔模型(DSSM)**

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| 协同过滤 | 简单 | 冷启动差、无法利用文本 | ❌ |
| DSSM双塔 | 用户塔+商品塔分离,可各自独立计算 | 交互信息有限 | ✅ 本项项目 |
| 深度交叉模型 | 精度高 | 无法独立建索引,召回耗时 | ❌ 排序层用 |

**工作原理**:
```
用户ID + 用户特征 → [用户塔] → user_embedding (128维向量)
                                          ↓ 余弦相似度
商品ID + 商品特征 → [商品塔] → item_embedding (128维向量)

FAISS: 把10万商品向量建索引 → 输入用户向量 → 返回Top200最相似商品
```

**FAISS 索引选择**:
- 10万级 → `IndexFlatIP` (内积索引,精确检索)
- 100万级 → `IndexIVFFlat` (倒排索引,近似检索)
- 线上部署 → `IndexHNSW` (图索引,最佳速度-精度平衡)

### 3.4 排序层 — 待实现

**多目标排序**:

| 目标 | 含义 | 为什么重要 |
|------|------|-----------|
| CTR (点击率) | 用户是否会点这个商品 | 曝光才有后续转化 |
| CVR (购买率) | 点进去后会不会买 | 直接影响 GMV |
| ATC (加购率) | 会不会加入购物车 | 强烈的购买意图信号 |

**加权融合公式**:
```
final_score = w1 × p_ctr + w2 × p_cvr + w3 × p_atc
           = 0.3 × p_ctr + 0.5 × p_cvr + 0.2 × p_atc
```
(权重可通过业务目标调整，如大促期间提高 CVR 权重)

### 3.5 重排层 — 待实现

| 策略 | 规则 | 目的 |
|------|------|------|
| 库存过滤 | 库存<阈值的商品降权或移除 | 避免推荐买不到的商品 |
| 新品加权 | 冷启动物品 (购买≤3次) 加权 15% | 给新品机会，均衡库存 |
| 多样性打散 | 同一品类最多出现3个 | 避免推荐列表全是同类商品 |
| 价格多样性 | 至少保留2个价格段的商品 | 覆盖用户不同预算需求 |
| 去重 | 过去7天已购买的商品降权 | 避免重复推荐已购商品 |

### 3.6 A/B 测试层 — 待实现

| 要素 | 方案 |
|------|------|
| 分流 | 用户ID哈希 → 50%对照组 + 50%实验组 |
| 对照组 | 热门商品兜底推荐 (按销量降序) |
| 实验组 | 完整推荐系统 (召回+排序+重排) |
| 核心指标 | CTR、CVR、人均GMV、次日留存 |
| 显著性检验 | t检验 / Bootstrap，p<0.05 |
| 实验周期 | 建议 14 天（覆盖完整周周期） |
| 多臂老虎机 | 后续：用MAB算法动态调整流量分配，自动收敛到最优策略 |
